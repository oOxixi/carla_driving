# Day23 模型选型与能力边界说明

## 1. 文档目的

本文档用于说明：

- 为什么项目使用 Qwen；
- Qwen 在系统中参与了哪些环节；
- Qwen 没有参与哪些环节；
- 为什么 Qwen 不直接控制车辆；
- Qwen 与 ONNX 检测模块的区别；
- 模型的已知能力和局限；
- 第一组答辩时应采用的统一表述。

## 2. 系统总体架构

本项目采用分层架构，而不是让一个大模型独立完成全部自动驾驶任务。

整体链路如下：

```text
驾驶员语音
→ 语音识别文本
→ RGB 图像与 SceneState
→ Qwen 高层语义理解与决策
→ 高层动作协议校验
→ 安全过滤
→ 车辆执行模块
→ 底层车辆控制
→ 最终安全仲裁
→ CARLA 车辆动作
```

主要模块包括：

### 2.1 语音识别模块

负责将驾驶员语音转换成自然语言文本。

### 2.2 视觉与环境感知模块

负责提供：

- RGB 图像；
- 目标检测结果；
- 车辆状态；
- 环境状态；
- SceneState。

### 2.3 Qwen 高层决策模块

负责：

- 理解驾驶员指令；
- 理解当前场景；
- 融合视觉信息和 SceneState；
- 生成高层驾驶动作；
- 输出简短决策依据。

### 2.4 车辆执行模块

负责将高层动作转换为：

- 目标速度；
- 目标车道；
- 停车目标；
- 避障目标；
- 其他车辆行为目标。

### 2.5 底层控制模块

负责计算实际车辆控制量，包括：

- `throttle`；
- `brake`；
- `steer`。

### 2.6 安全仲裁模块

负责检查控制结果是否安全，并在危险情况下覆盖不安全决策。

## 3. 为什么选择 Qwen

本项目需要同时处理：

- 自然语言驾驶指令；
- RGB 图像；
- 结构化车辆状态；
- 环境目标；
- 多模态信息之间的语义关联；
- 固定协议下的高层动作输出。

Qwen 多模态模型适合承担高层语义理解任务。

例如驾驶员说：

```text
前面那辆车太慢了，找机会绕过去。
```

系统需要理解：

- “前面那辆车”对应哪个场景目标；
- 当前是否存在前车；
- 相邻车道是否可用；
- 当前场景是否适合变道；
- 驾驶员指令是否与安全状态冲突；
- 最终应变道、减速还是保持当前状态。

传统规则可以执行明确的安全约束，但很难独立处理自然语言中的省略、歧义和上下文关联。

因此，Qwen 用于完成高层理解和决策。

## 4. 为什么不让 Qwen 直接控制车辆

Qwen 不适合直接输出方向盘、油门和制动，主要原因如下：

1. 大模型输出具有概率性；
2. 大模型不能保证每次生成完全相同的连续控制值；
3. 大模型可能输出格式错误；
4. 大模型可能产生幻觉；
5. 大模型可能生成不符合车辆动力学约束的结果；
6. 大模型推理延迟高于传统车辆控制器；
7. 车辆底层控制需要高频、稳定和确定；
8. 安全关键控制需要可验证；
9. 大模型不能绕过安全仲裁；
10. 大模型不适合独立承担实时连续控制。

因此，本项目将 Qwen 限制为高层决策模块。

正确分工是：

```text
Qwen 决定“做什么”
车辆执行模块决定“转换为什么目标”
底层控制模块决定“具体怎样控制”
安全模块决定“最终是否允许执行”
```

## 5. Qwen 的输入

Qwen 高层决策模块接收：

1. 驾驶员自然语言指令；
2. RGB 摄像头图像；
3. SceneState；
4. 自车速度；
5. 自车位置；
6. 自车车道；
7. 周围车辆、行人和障碍物；
8. 目标距离和方向；
9. 道路和环境状态；
10. 安全相关场景信息。

Qwen 不直接读取方向盘、油门或制动执行器作为连续控制输入。

## 6. Qwen 的输出

Qwen 使用统一 JSON 格式输出高层驾驶意图：

```json
{
  "actions": [
    {
      "action": "SET_SPEED",
      "target_id": "",
      "target_speed_kmh": 0.0
    }
  ],
  "confidence": 0.0,
  "reason": ""
}
```

字段说明：

- `actions`：高层动作列表；
- `action`：动作白名单中的高层动作；
- `target_id`：SceneState 中真实存在的目标；
- `target_speed_kmh`：目标速度；
- `confidence`：模型对当前判断的置信度；
- `reason`：简短的决策依据。

`reason` 用于向开发人员和答辩人员解释决策依据，不要求模型输出内部逐步推理过程。

## 7. Qwen 允许输出的动作

Qwen 只能输出以下动作：

```text
START
STOP
SET_SPEED
TURN_LEFT
TURN_RIGHT
CHANGE_LANE_LEFT
CHANGE_LANE_RIGHT
AVOID_OBJECT
EMERGENCY_BRAKE
RETURN_TO_LANE
```

这些动作属于高层驾驶意图，不是底层车辆控制值。

## 8. Qwen 禁止输出的内容

Qwen 禁止输出：

```text
throttle
brake
steer
方向盘角度
油门值
制动力
```

Qwen 也不得：

- 绕过车辆执行模块；
- 绕过安全仲裁模块；
- 声称高层动作已经直接控制车辆；
- 将自身描述为完整端到端自动驾驶系统；
- 编造 SceneState 中不存在的目标；
- 在视觉证据不足时编造观察结果。

## 9. 高层动作示例

### 9.1 前车减速场景

驾驶员指令：

```text
前方车辆减速，请降低速度并保持安全距离。
```

Qwen 输出示例：

```json
{
  "actions": [
    {
      "action": "SET_SPEED",
      "target_id": "vehicle_12",
      "target_speed_kmh": 10.0
    }
  ],
  "confidence": 0.94,
  "reason": "前方车辆距离较近且速度较低，需要降低目标速度"
}
```

Qwen 只输出目标速度。

Qwen 不负责决定具体刹车力度。

后续由车辆执行模块和底层控制模块完成：

```text
SET_SPEED
→ 目标速度 10 km/h
→ 底层速度控制
→ 计算 throttle 和 brake
→ 安全仲裁
→ 最终车辆动作
```

### 9.2 紧急碰撞风险场景

Qwen 输出示例：

```json
{
  "actions": [
    {
      "action": "EMERGENCY_BRAKE",
      "target_id": "vehicle_12",
      "target_speed_kmh": 0.0
    }
  ],
  "confidence": 0.98,
  "reason": "前方存在明确紧急碰撞风险"
}
```

即使 Qwen 输出 `EMERGENCY_BRAKE`，最终制动力仍由车辆控制和安全模块决定。

## 10. 安全优先原则

当驾驶员指令与安全状态冲突时，安全优先。

例如驾驶员发出指令：

```text
继续加速。
```

但系统检测到：

- 前方车辆距离过近；
- 前方为红灯；
- 行人进入道路；
- 停止线距离过近；
- TTC 低于安全阈值；
- 存在明确碰撞风险。

此时系统不应继续执行加速。

理想情况下，Qwen 应输出减速、停车或紧急制动等安全动作。

即使 Qwen 输出了不安全的高层动作，安全模块仍可以覆盖该动作。

安全模块覆盖 Qwen 不代表系统失败，而是分层安全设计正常生效。

## 11. 低置信度处理

当出现以下情况时，Qwen 应降低 `confidence`：

- 驾驶员指令含糊；
- RGB 图像缺失；
- 图像内容不清晰；
- SceneState 信息不完整；
- 图像和 SceneState 不能互相印证；
- 无法确定驾驶员指代的具体目标；
- 无法确认变道或避障条件；
- 场景中存在相互冲突的信息。

低置信度时，系统不应盲目执行激进动作。

建议行为包括：

- 保持当前安全速度；
- 降低目标速度；
- 等待更多场景信息；
- 交给安全模块进一步判断。

## 12. Qwen 的已知局限

Qwen 可能出现以下问题：

- 误解自然语言指令；
- 对含糊指令进行错误推断；
- 错误关联图像目标和 SceneState 目标；
- 对视觉目标产生误识别；
- 在证据不足时产生幻觉；
- 输出不合法 JSON；
- 输出白名单之外的动作；
- 输出不存在的 `target_id`；
- 输出空动作列表；
- 置信度与真实正确率不完全一致；
- 推理延迟较高；
- 不适合承担高频底层控制；
- 相同输入可能出现一定输出差异。

## 13. 项目如何降低模型风险

本项目采用以下措施约束 Qwen：

- 固定最终提示词；
- 固定动作白名单；
- 固定 JSON 输出结构；
- 拒绝非法动作；
- 拒绝空动作结果；
- 限制 `confidence` 范围；
- 校验 `target_id`；
- 禁止输出底层控制字段；
- 增加安全过滤；
- 使用车辆执行模块转换目标；
- 使用确定性底层控制器；
- 保留安全模块最终仲裁权；
- 保存完整决策链；
- 检查 Qwen 判断与最终车辆动作是否一致。

## 14. Qwen 与 ONNX 检测模块的区别

ONNX 是一种模型交换和推理格式。

项目中的 ONNX 检测模块通常负责：

- 检测目标；
- 输出目标类别；
- 输出目标位置；
- 输出边界框；
- 输出检测置信度；
- 提供结构化视觉结果。

ONNX 检测模块不是视觉语言大模型。

Qwen 负责：

- 理解自然语言；
- 理解图像语义；
- 融合 RGB 图像与 SceneState；
- 进行场景关联；
- 生成高层驾驶动作；
- 输出简短决策依据。

因此：

```text
ONNX 检测模块负责“看到了什么”
Qwen 负责“这些信息意味着什么，以及高层应该做什么”
```

不能把 ONNX 检测模块描述成 Qwen。

也不能把 Qwen 描述成底层目标检测器。

## 16. 模型参与了哪些环节

答辩时可以回答：

Qwen 参与了三个主要环节：

1. 驾驶员自然语言指令理解；
2. RGB 图像和 SceneState 的多模态场景理解；
3. 有限动作集合内的高层驾驶决策生成。

Qwen 输出统一的 DrivingIntent，再交给车辆执行模块处理。

## 17. 模型没有参与哪些环节

答辩时可以回答：

Qwen 没有直接参与：

- ONNX 检测模型的底层推理过程；
- 方向盘控制计算；
- 油门控制计算；
- 制动控制计算；
- 车辆动力学控制；
- 高频闭环控制；
- 最终安全仲裁。

## 18. 为什么不是端到端自动驾驶

答辩时可以回答：

车辆底层控制要求高频、稳定、确定、实时和可验证。

多模态大模型更适合自然语言理解、视觉语义关联和高层行为决策，不适合直接承担安全关键的连续控制。

因此，本项目采用分层架构：

```text
Qwen 高层决策
+ 车辆执行模块
+ 确定性底层控制器
+ 安全仲裁模块
```

这比让大模型直接控制车辆更安全，也更容易测试和解释。
